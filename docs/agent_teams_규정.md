# U2DIA AI Agents — 팀 운영 헌법

**Version**: 4.0.0
**Last Updated**: 2026-02-26
**철학**: 에이전트의 능력을 100% 발휘하게 하되, 팀이 동작하기 위한 최소한의 원칙만 정의한다.

---

## 핵심 사상

> **규제가 아닌 헌법.** 에이전트에게 "어떻게"를 지시하지 않는다. "무엇을 지켜야 하는가"만 정의한다.
> 에이전트는 자신의 전문성과 판단으로 최선의 방법을 선택할 자유가 있다.

이전 버전(v3.0)은 메시지 타입 4가지, 산출물 타입 5가지, 로그 이벤트 6가지, 30분/15분 시간 규제, 역할별 산출물 형식 등을 규정했다. **이 모든 규제를 폐지한다.** 에이전트는 결과로 평가받는다.

## 프로젝트 그룹 (project_group)

> **project_group = git 프로젝트 폴더명** (MCP 토큰 인증 시 자동 설정)

- 각 프로젝트는 고유한 MCP 토큰으로 인증되며, 토큰에 연결된 프로젝트명이 `project_group`으로 자동 설정됨
- 하나의 프로젝트 그룹 아래 여러 팀이 존재할 수 있음
- 대시보드에서 프로젝트별로 그룹핑되어 가시성 확보

## 아키텍처

  │  │         │         │              │

  └──────────────────────────────────────┘

**역할은 고정이 아니다.** Backend, Frontend, QA, DevOps는 예시일 뿐이다. 오케스트레이터는 프로젝트에 필요한 어떤 역할이든 자유롭게 정의하고 스폰한다.

## 불변 원칙 (4가지만)

### 1. 컨텍스트 격리 (Context Isolation)

각 에이전트는 독립된 세션에서 작업한다. 다른 에이전트의 컨텍스트에 직접 접근하지 않으며, 칸반보드의 메시지와 산출물을 통해 소통한다.

### 2. 티켓 선행 (Ticket-First)

코딩 전에 반드시 티켓이 존재해야 한다. 티켓 없는 작업은 추적 불가능하므로, 존재하지 않는 것과 같다.

### 3. 오케스트레이터 조율 (Orchestrator Coordination)

- 오케스트레이터만 팀을 생성하고, 티켓을 분할하고, 에이전트를 스폰한다
- 에이전트는 배정된 티켓 범위 내에서 자유롭게 작업한다
- 오케스트레이터는 결과를 검증하고, 필요 시 개입한다

### 4. 기록 의무 (Record Obligation)

모든 작업의 흔적이 칸반보드에 남아야 한다:
- **진행**: 액티비티 로그 (형식 자유)
- **소통**: 메시지 (타입 자유)
- **결과**: 산출물 (형식 자유)

## 워크플로우 (간소화)

### 오케스트레이터

```
1. 프로젝트 분석 → 팀 생성
2. 업무 원자화 → 티켓 등록 (의존성/우선순위 포함)
3. 에이전트 스폰 → 작업 지시 (team_id, member_id, ticket_ids만 전달)
4. 모니터링 → 필요 시 개입
5. 검증 → 피드백 → 최종 승인
```

### 에이전트

```
1. 티켓 점유 (claim)
2. 작업 수행 (방법은 자유)
3. 결과 등록 (산출물 + 로그)
4. 완료 전환 (→ Review)
```

**2번과 3번 사이에 무엇을 하든 에이전트의 자유다.** 다른 에이전트에게 질문할 수도, 코드 리뷰를 요청할 수도, 혼자 해결할 수도 있다. 순서도 강제하지 않는다.

---

## 티켓 상태 흐름

```
Backlog → Todo → InProgress → Review → Done
                      ↓                  ↓
                   Blocked          InProgress (반려)
```

이것이 유일한 프로세스 규정이다. 상태 전이 순서만 지키면 된다.

---

## 의존성

`depends_on`으로 지정된 선행 티켓이 Done이 아니면 claim 불가 (시스템이 자동 거부).

```
Schema 완료 → Service/API 가능
API 완료    → Frontend 연동 가능
구현 완료   → 테스트 가능
```

---

## 오케스트레이터 개입 기준

오케스트레이터는 **결과 기반**으로 판단한다. 시간 기반 규제(30분, 15분)는 폐지한다.

| 상황 | 판단 |
|------|------|
| 의존성 교착 | 우선순위 재조정 |
| 반복 실패 | 에이전트 교체 또는 직접 개입 |
| 빌드 실패 | 관련 에이전트에게 수정 지시 |
| Blocked 누적 | 구조적 문제 분석 → 티켓 재구성 |

**개입은 최후의 수단이다.** 에이전트가 스스로 해결할 시간과 자유를 먼저 부여한다.

---

## 피드백

완료된 티켓에 대해 피드백(1-5점)을 등록할 수 있다. 피드백은 다음 작업의 개선에 활용한다. 카테고리는 자유롭게 정의 가능.

---

## v3.0에서 폐지된 규제

| 폐지 항목 | 이유 |
|-----------|------|
| 메시지 타입 강제 (comment/question/code_review/reply) | 에이전트가 자유롭게 소통 |
| 산출물 타입 강제 (code/file_path/result/summary/log) | 어떤 형태든 결과물 등록 |
| 로그 이벤트 강제 (6가지) | 자유로운 이벤트 기록 |
| 역할별 산출물 형식 (Backend=code+file_path 등) | 에이전트가 최적 형식 결정 |
| 30분 무활동 재배정 | 시간 규제 → 결과 기반 평가 |
| 15분 응답 의무 | 에이전트 자율 판단 |
| 피드백 카테고리 고정 (4가지) | 자유 카테고리 |
| 역할 고정 (Backend/Frontend/DB/QA/DevOps) | 자유 역할 정의 |

## v4.1 보강 규정 (2026-04-04)

> v4.0의 자유 원칙은 유지하되, 좀비 티켓·무보고 잠수·범위 이탈을 방지하는 **최소 enforcement**를 추가한다.

### 5. 전문 에이전트 신원 (Agent Identity)

> **모든 에이전트는 이름이 부여된 전문가로 스폰된다.**

- 스폰 시 고유 이름 배정 (Alex, Blake, Casey, Dana, Ellis, Finn, Gray, Harper, Ivy, Jules, Kit, Logan, Max, Noel, Owen, Parker, Quinn, Riley, Sam, Tay, Uma, Val, Wren, Xia, Yuri, Zane, Ash, Brook, Clay, Drew)
- 이름은 member_id 해시로 고정 (재접속해도 같은 이름)
- **역할(role)에 해당하는 티켓만 클레임** — 서버가 역할-티켓 매칭 검증
- 그레이존 → supervisor가 범용 에이전트 호출 또는 가장 가까운 전문가 지정
- Agent Office에서 캐릭터로 시각화 (288가지 외모 조합)

### 6. 중간 보고 의무 (Progress Reporting)

> **InProgress 전환 후 반드시 progress_note를 등록해야 한다.**

- 티켓 claim → InProgress 전환 시, **최초 progress_note 등록 의무**
- 이후 의미 있는 진행이 있을 때마다 업데이트
- **Supervisor 순회 점검 (5분 간격)**: InProgress인데 progress_note 없으면 경고 발송
- 경고 2회 무시 시 → **자동 unclaim + Backlog 전환**

```
claim → InProgress → progress_note 등록 → 작업 → 산출물 → Review
                    ↑ 여기가 필수
```

### 6. 역할 범위 강제 (Scope Enforcement)

> **에이전트는 배정된 티켓 범위 내에서만 작업한다. 범위 밖 작업이 필요하면 supervisor를 호출한다.**

- 자기 팀의 티켓만 claim 가능 (서버에서 team_id 검증)
- 다른 팀의 코드/파일을 수정해야 할 경우 → supervisor에게 메시지로 요청
- supervisor가 관련 팀에 새 티켓을 발행하거나, 에이전트 간 회의를 주선

### 7. 에이전트 간 소통 프로토콜 (Inter-Agent Communication)

> **에이전트 간 직접 소통은 칸반 메시지로 기록한다. 회의가 필요하면 supervisor를 경유한다.**

- 질문: 칸반 메시지 (`message_type: question`)로 등록 → 대상 에이전트가 응답
- 회의: supervisor에게 회의 요청 → supervisor가 회의 티켓 생성 + 참여 에이전트 소환
- 코드 리뷰: 산출물 등록 후 메시지로 리뷰 요청 → 피드백은 칸반에 기록
- **DM 금지**: 칸반 외부에서의 에이전트 간 직접 통신은 기록되지 않으므로 금지

### 8. 무활동 자동 해제 (Idle Auto-Release)

> **InProgress 상태에서 장기 무활동 시 자동으로 티켓이 해제된다.**

| 경과 시간 | 조치 |
|-----------|------|
| 10분 무보고 | 1차 경고 (팀 메시지) |
| 20분 무보고 | 2차 경고 + supervisor 알림 |
| 30분 무보고 | 자동 unclaim → Todo로 복귀 |

### 9. 산출물 필수 검증 (Artifact Mandatory)

> **Review 전환 시 산출물이 1개 이상 등록되어 있어야 한다.**

- 산출물 없이 Review 전환 시도 → 서버에서 거부
- Supervisor QA에서도 산출물 없으면 자동 1점 (rework)
- 산출물 내용이 50자 미만이면 "형식적 산출물"로 간주 → 2점 이하

### 10. 재작업 한도 (Rework Limit)

> **동일 티켓의 재작업은 최대 3회로 제한한다.**

- 1~2회: InProgress로 복귀, 피드백 반영 후 재시도
- 3회 도달: **Blocked 에스컬레이션** → supervisor + 대표에게 알림
- Blocked 해소: 대표 또는 supervisor가 직접 판단하여 재구성
